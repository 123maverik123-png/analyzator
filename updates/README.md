# Анализатор звонков техподдержки

Распознаёт записи разговоров (Whisper), разделяет голоса оператора и абонента
(диаризация), анализирует тональность (rubert) и смысл (YandexGPT: проблема,
категория, решено/нет, действие, резюме), а метрики по отделу собирает в
центральный дашборд для руководителя.

## Архитектура

Децентрализованная: бот стоит на ПК каждого оператора, обрабатывает записи **локально**.
Аудио и полный текст разговора **не покидают ПК оператора**. В центр по сети
(WireGuard) уходят только метрики (оператор, дата, категория, решено, тональность).

```
ПК оператора (клиент)                Машина-центр (server/)
запись → Whisper → диаризация        приёмник FastAPI → центральная БД → дашборд
   → анализ → локальный SQLite          ▲
   → очередь отправки  ── метрики ──────┘  (HTTPS + общий токен, только в WireGuard-сети)
```

## Диаризация (разделение голосов)

Записи **моно** (оператор и абонент в одной дорожке), поэтому разделение делается
по тембру голоса через `pyannote.audio`, а не по аудиоканалам. После распознавания
Whisper каждый сегмент текста сопоставляется со спикером диаризации по перекрытию
времени; первый говорящий считается оператором (эвристика — точного способа узнать
роль из аудио нет).

Чтобы включить:
1. Создайте токен на [hf.co/settings/tokens](https://hf.co/settings/tokens).
2. Примите условия доступа на странице модели
   [pyannote/speaker-diarization-community-1](https://huggingface.co/pyannote/speaker-diarization-community-1).
3. Впишите токен в `secret.env` как `HF_TOKEN`.

Если `HF_TOKEN` не задан, модель не скачалась (нет сети) или `DIARIZATION_ENABLED=0` —
бот не падает, а просто работает как раньше, без разделения голосов (`diarized: false`
в отчёте). В отчётах и БД при успешной диаризации появляются поля `dialogue_text`
(текст с метками `[Оператор]`/`[Абонент]`) и `turns` (реплики с таймингами); сам
смысловой LLM-анализ тоже использует размеченный диалог, если он доступен — это
помогает точнее определить, кто и что сделал для решения проблемы.

## Установка клиента (на ПК оператора)

1. Python 3.12, затем:
   ```bash
   pip install -r requirements.txt
   ```
2. Скопируйте `secret.env.example` → `secret.env` и заполните `YC_API_KEY`,
   `YC_FOLDER_ID`, `HF_TOKEN` (для диаризации), `OPERATOR_ID`, `CENTRAL_URL`, `CENTRAL_TOKEN`.
3. Укажите папку с записями в `config.py` (`WATCH_FOLDER`).
4. Запуск: `start_bot.bat` (консоль) или `start_all.bat` (GUI).

Если `CENTRAL_URL` пустой — клиент работает автономно (только локальная БД).
При оффлайне метрики копятся и досылаются автоматически, когда связь появится.

## Установка центра (одна машина в WireGuard-сети)

```bash
cd server
pip install -r requirements.txt
set CENTRAL_TOKEN=длинный_случайный_секрет_тот_же_что_у_клиентов
set DASH_USER=user            # логин для входа в дашборд (по умолчанию user)
set DASH_PASSWORD=свой_надёжный_пароль
uvicorn server:app --host 0.0.0.0 --port 8000
```

Дашборд для руководителя: `http://<wireguard-ip>:8000/` — при открытии браузер
спросит логин и пароль (`DASH_USER` / `DASH_PASSWORD`). Выгрузка в Excel: кнопка
«Скачать CSV» на странице. Приём метрик от ботов (`/api/results`) защищён отдельно —
токеном `CENTRAL_TOKEN`.

## Файлы

| Файл | Назначение |
|------|------------|
| `main.py` | наблюдение за папкой, оркестрация обработки |
| `transcriber.py` | распознавание речи (faster-whisper) + диаризация (pyannote.audio) |
| `analyzer.py` | тональность (rubert) + разбор через YandexGPT |
| `storage.py` | локальная БД звонков на ПК оператора |
| `sender.py` | отправка метрик в центр с очередью |
| `gui.py` | локальный GUI оператора |
| `server/server.py` | центральный приёмник + дашборд |
